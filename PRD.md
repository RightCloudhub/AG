## 1. 产品概述

### 1.1 产品定位

面向知识密集型业务场景的**图谱增强多跳推理智能问答系统**。系统以知识图谱为"结构化记忆"、以 Agent 为"决策大脑"，通过"问题分解 → 图上多跳检索 → 反思校验 → 迭代追问"的闭环，解决传统向量 RAG 无法回答的跨实体、跨文档、多跳关系类问题，并输出可审计的推理链。

### 1.2 要解决的核心问题

| 问题 | 现状 | 本产品的解法 |
|---|---|---|
| 多跳关系问题答不出/答不准 | 向量检索只召回语义相似片段，拼不出推理链 | Planner 分解子问题，Agent 沿图谱按需多跳检索 |
| 检索过程黑盒、无法审计 | 相似度打分不可解释 | 强制输出结构化推理链（子问题→图路径→证据→结论） |
| 检索策略静态、不会"追问" | 固定一轮检索即生成 | Critic 判断证据充分性，不足则触发下一跳 |
| 复杂问题与简单问题同等对待 | 所有请求走同一管线 | 复杂度分诊：简单问题走 Fast Path，复杂问题走 Agentic 流程 |

### 1.3 非目标（Out of Scope，V1）

- 不做开放域全网问答，仅覆盖已接入的领域语料。
- 不做多模态（图片/表格图像）理解，V1 仅处理文本类文档。
- 不做面向终端消费者的开放注册产品，V1 面向企业内部用户与系统集成方。
- 不承诺实时（秒级）图谱更新，V1 增量更新为准实时（分钟~小时级）。

## 2. 用户与场景

### 2.1 目标用户画像

| 角色 | 描述 | 核心诉求 |
|---|---|---|
| **业务分析用户** | 投研/风控/合规/情报分析人员 | 提出复杂关系问题，获得准确答案 + 可追溯依据 |
| **业务系统集成方** | 内部应用/机器人的开发团队 | 稳定的 API、可预期的延迟与成本、清晰的错误语义 |
| **知识运营人员** | 负责语料接入与图谱质量 | 便捷的文档接入、抽取质量审核、实体消歧工单 |
| **系统管理员** | 平台运维 | 监控、配额、预算控制、审计日志 |

### 2.2 典型用户故事

1. 作为**分析师**，我提问"A公司的母公司的CEO曾在哪些公司任职？"，系统应给出答案，并展示"A公司→母公司→CEO→任职历史"的图路径与引用来源，以便我核实。
2. 作为**分析师**，我提问"产品X的供应商中，哪些同时也是竞品B的供应商？"，系统应基于图结构做交集计算，而非拼接文本片段。
3. 作为**分析师**，当证据不足时，我希望系统明确回答"未找到充分证据"并列出已探索的路径，而不是编造答案。
4. 作为**集成方**，我调用查询 API 时能拿到结构化的答案、推理链、引用与置信度，并能通过参数控制最大跳数与超时。
5. 作为**知识运营**，我上传一批新文档后，系统自动抽取实体关系并入图，冲突项进入人工审核队列，无需全量重建。
6. 作为**管理员**，我能看到每次查询的跳数、LLM 调用次数、token 成本，并能对租户/用户设置预算上限。

## 3. 功能需求

需求优先级定义：**P0** = MVP 必须，**P1** = 试点上线前必须，**P2** = 规模化阶段。

### 3.1 知识层：图谱构建与管理

| 编号 | 需求 | 描述 | 优先级 |
|---|---|---|---|
| FR-KG-01 | 文档接入 | 支持批量导入文本类文档（Markdown/TXT/PDF文本/HTML），记录来源元数据（出处、时间、版本） | P0 |
| FR-KG-02 | 实体/关系抽取 | LLM 结构化抽取三元组（实体-关系-属性），辅以规则/词典兜底；每条三元组携带置信度与来源引用 | P0 |
| FR-KG-03 | Schema 管理 | 支持定义/查看领域 Schema（实体类型、关系类型、属性约束）；抽取结果须符合 Schema 校验 | P0 |
| FR-KG-04 | 实体消歧 | 同名/别名实体的消歧与合并（Entity Resolution）；低置信度合并进入人工审核队列 | P1 |
| FR-KG-05 | 增量更新 | 新文档抽取后经"冲突检测→合并入图"流程增量写入，不中断线上查询，不做全量重建 | P1 |
| FR-KG-06 | 质量治理 | 抽取抽样审核界面：人工标注对/错，形成质量闭环；支持按置信度阈值过滤入图 | P1 |
| FR-KG-07 | 图谱浏览 | 提供实体详情页与邻居子图可视化，供运营排查 | P2 |

### 3.2 检索与推理层：混合检索

| 编号 | 需求 | 描述 | 优先级 |
|---|---|---|---|
| FR-RT-01 | 向量检索 | 文档 chunk 级 embedding 语义召回，Top-K 可配置 | P0 |
| FR-RT-02 | 图检索 | 支持多跳邻居查询、路径查询、子图遍历（Cypher/GQL 封装为检索工具），带相关性剪枝与 Top-K 路径采样 | P0 |
| FR-RT-03 | 全文检索 | BM25 关键词检索，覆盖专有名词与精确匹配场景 | P0 |
| FR-RT-04 | 融合排序 | RRF 融合三路结果为统一候选列表；预留学习型 Re-ranker 接口 | P1 |
| FR-RT-05 | 检索并行化 | 同一子问题的多路检索并行执行 | P1 |

### 3.3 Agent 编排层（核心）

| 编号 | 需求 | 描述 | 优先级 |
|---|---|---|---|
| FR-AG-01 | 复杂度分诊 | 查询入口判定问题复杂度：简单问题走 Fast Path（单轮向量RAG+生成）；复杂问题进入 Agentic 流程 | P1 |
| FR-AG-02 | Planner 问题分解 | 将复杂问题分解为子问题序列，支持树状/图状依赖（非仅链式）；子问题可依赖前序结果动态生成 | P0 |
| FR-AG-03 | Executor 工具编排 | 针对子问题选择检索工具（向量/图/全文，预留外部 SQL/API 工具接口）并执行 | P0 |
| FR-AG-04 | Critic 反思校验 | 判定当前证据是否足以回答子问题/原问题；不足则触发下一跳检索或改写子问题 | P0 |
| FR-AG-05 | Memory 状态管理 | 维护已探索路径、已获证据、已排除假设；同一路径不重复检索，防止死循环 | P0 |
| FR-AG-06 | 循环护栏 | 最大跳数上限、最大 LLM 调用次数、单查询 token 预算上限均可配置；触顶时走"未找到答案"兜底并返回已有证据 | P0 |
| FR-AG-07 | 诚实兜底 | 证据不足时明确输出"无法基于现有知识回答"，并附已探索路径摘要；禁止无引用的编造性回答 | P0 |
| FR-AG-08 | 外部工具扩展 | 支持注册外部工具（SQL 查询、内部 API）供 Executor 调用，含权限声明 | P2 |

### 3.4 答案生成与可解释性

| 编号 | 需求 | 描述 | 优先级 |
|---|---|---|---|
| FR-AN-01 | 答案生成 | 基于图路径 + 文本证据生成最终答案；答案中的关键论断须绑定引用 | P0 |
| FR-AN-02 | 结构化推理链 | 输出机器可读的推理链：子问题 → 使用的工具 → 命中的节点/边/文档片段 → 中间结论 → 最终答案 | P0 |
| FR-AN-03 | 推理链可视化 | 前端以"分解树 + 图路径 + 引用列表"形式可视化推理链，节点可点击查看原文 | P1 |
| FR-AN-04 | 推理链落库 | 每次查询的完整推理链持久化存储，支持按查询 ID 审计回查 | P1 |
| FR-AN-05 | 置信度输出 | 答案附整体置信度分级（高/中/低），依据证据覆盖度与路径置信度计算 | P2 |

### 3.5 服务接口

| 编号 | 需求 | 描述 | 优先级 |
|---|---|---|---|
| FR-API-01 | 查询 API | `POST /v1/query`：入参含问题、可选参数（最大跳数、超时、是否强制 Agentic）；出参含答案、推理链、引用、耗时与成本统计 | P0 |
| FR-API-02 | 流式响应 | 支持 SSE 流式返回中间进度（当前子问题、当前跳数）与最终答案，改善长耗时体验 | P1 |
| FR-API-03 | 知识管理 API | 文档上传、抽取任务状态查询、审核队列操作 | P1 |
| FR-API-04 | 统一响应封装 | 所有 API 使用统一 envelope：success 标志、data、error、分页 meta | P0 |
| FR-API-05 | 问答 Web 界面 | 内部试用界面：提问、查看答案与推理链可视化、反馈按钮（准确/不准确） | P1 |

### 3.6 运营与管理

| 编号 | 需求 | 描述 | 优先级 |
|---|---|---|---|
| FR-OP-01 | 查询监控 | 按查询维度记录：延迟分布、跳数分布、LLM 调用次数、token 成本、工具调用明细 | P1 |
| FR-OP-02 | 预算控制 | 按租户/用户/单查询设置 LLM 成本上限，超限熔断并返回明确错误 | P1 |
| FR-OP-03 | 用户反馈回路 | 收集"准确/不准确"反馈并关联推理链，形成 badcase 库供迭代 | P1 |
| FR-OP-04 | 评测集与回归 | 自建多跳评测集（≥200条，分2跳/3跳/开放路径分级）；提供一键评测脚本输出 Accuracy/Recall 对比报告 | P0 |

## 4. 非功能需求

| 编号 | 类别 | 需求 |
|---|---|---|
| NFR-01 | 性能 | 单次查询端到端延迟：Fast Path P95 ≤ 3s；Agentic 流程 P95 ≤ 8s（具体阈值随业务场景在评审会确认） |
| NFR-02 | 效果 | 多跳评测集 Answer Accuracy 较纯向量 RAG Baseline 提升 ≥ 25 个百分点；跳数 ≥2 的问题证据 Recall ≥ 85% |
| NFR-03 | 可用性 | 图谱增量更新期间线上查询不中断；查询服务可用性 ≥ 99.5%（试点阶段） |
| NFR-04 | 成本 | 单次 Agentic 查询 LLM 成本有硬上限（可配置）；复杂度分诊使简单问题不进入多轮循环 |
| NFR-05 | 可解释性 | 100% 的 Agentic 查询产出结构化推理链并可审计回查 |
| NFR-06 | 安全 | 所有密钥经环境变量/密钥管理注入，不硬编码；API 鉴权 + 速率限制；文档与图谱数据按租户隔离；错误信息不泄露内部细节 |
| NFR-07 | 输入校验 | 所有外部输入（用户问题、上传文档、API 参数）经 schema 校验；防注入（含 Cypher/SQL 注入与 Prompt 注入基础防护） |
| NFR-08 | 可观测性 | 全链路 trace（查询→子问题→工具调用→LLM调用），日志含查询 ID 可串联 |
| NFR-09 | 可测试性 | 单测/集成/E2E 覆盖率 ≥ 80%；评测集回归纳入 CI |
| NFR-10 | 扩展性 | 图数据库、向量库、LLM 供应商均通过抽象接口接入（Repository 模式），支持替换选型 |

## 5. 系统架构（摘要）

四层架构（详见立项建议书 4.1 架构图）：

1. **接入层**：查询 API / Web 界面 / 流式通道。
2. **Agent 编排层**：Planner、Executor、Critic、Memory 四组件闭环；复杂度分诊器位于入口。
3. **检索与推理层**：向量检索、图检索、全文检索三路并行 + RRF 融合排序。
4. **知识层**：知识图谱（Neo4j/NebulaGraph 选型待定）、向量库、原始文档库；配套抽取管线与增量更新流程。

关键设计原则：
- 图谱不直接给答案，为 Agent 提供多跳导航能力；
- 检索跳数不预设，由 Critic 动态决定，护栏兜底；
- 所有循环有预算与上限，成本可控。

## 6. 关键流程

### 6.1 Agentic 查询主流程

1. 用户提交问题 → 复杂度分诊。
2. 简单问题：向量检索 → 生成答案（附引用）→ 返回。
3. 复杂问题：Planner 分解子问题 → Executor 逐个执行（选择工具、并行三路检索、融合排序）→ Critic 判定证据充分性。
4. 不充分且未触顶：改写/追加子问题，回到步骤 3（下一跳）。
5. 充分或触顶：生成最终答案 + 结构化推理链；触顶未答出则返回诚实兜底。
6. 推理链落库，记录成本与延迟指标。

### 6.2 知识增量更新流程

新文档上传 → 抽取三元组（带置信度）→ Schema 校验 → 与存量图冲突检测 → 高置信自动合并 / 低置信进入人工审核队列 → 审核通过入图 → 索引（向量/全文）同步更新。

## 7. 验收标准

与立项建议书第九节对齐，量化为可执行验收项：

| 编号 | 验收项 | 通过标准 | 验证方式 |
|---|---|---|---|
| AC-1 | 多跳准确率 | 评测集（≥200条）Accuracy 较 Baseline 提升 ≥25pp | 一键评测脚本输出对比报告 |
| AC-2 | 证据召回 | 跳数≥2 问题的支持证据 Recall ≥85% | 评测脚本，人工标注金标证据 |
| AC-3 | 推理链 | 100% Agentic 查询产出可视化推理链且可按查询 ID 回查 | 抽样审计 |
| AC-4 | 延迟 | P95 达到约定阈值（Agentic ≤8s 目标） | 压测报告 |
| AC-5 | 增量更新 | 新增文档入图期间线上查询无中断、无需全量重建 | 演练验证 |
| AC-6 | 护栏 | 构造发散问题，验证最大跳数/预算触顶后正确兜底、成本不超限 | 专项测试用例 |
| AC-7 | 诚实性 | 构造无答案问题集，编造率（无引用断言）为 0 | 专项评测 |

## 9. 风险与开放问题

风险沿用立项建议书第八节。评审会需确认的开放问题：

1. **图数据库选型**：Neo4j（生态成熟）vs NebulaGraph（分布式、大规模）——取决于数据规模与团队熟悉度。
2. **延迟阈值**：P95 ≤ 8s 是目标值，需与首个试点业务方确认可接受范围。
3. **评测集标注资源**：≥200 条多跳问题的金标证据标注人力来源（人工 vs 合成+抽检）。
4. **首个试点领域**：选定单一高价值领域作为 POC/MVP 语料范围。
5. **LLM 选型与预算**：Agent 各角色（Planner/Executor/Critic）是否分级用模（强模型规划、轻模型执行）以控成本。
