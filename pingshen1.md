## 一、执行摘要

本项目在**技术路线定义与工程骨架**上明显优于常见“GraphRAG 概念包装”型方案：问题定义清晰，图谱/向量/Agent 分工明确，已落地 Planner→Executor→Critic→Answer 闭环、结构化推理链契约、三路检索抽象、护栏与预算、以及可复现评测流水线。G2 离线 dev 集（153 题）报告 Accuracy 约 **60.8%**、相对纯向量 Baseline **+49pp**、证据 Recall **0.88**，说明在**合成公司关系图谱 + 离线确定性答案器**条件下，多跳路径检索具备可验证优势。

最大亮点是：**把“多跳、可审计、诚实拒答、门禁止损”做成了可运行系统与治理机制**，而非仅停留在架构图。最大风险则是**场景与证据的双重错位**：当前 Schema 与语料为合成“公司/人员/产品/事件”关系域，并非制度、合同、工单、流程等真实企业内知识；实体消歧、增量更新、权限隔离、复杂度分诊、RRF 融合、推理链落库等企业级关键能力均排至阶段三/四或未实现；G1/G2 的高分数字主要依赖离线启发式与合成金标，**不能**等同生产 LLM 路径或真实业务效果。

**综合判定**：技术路线值得继续投入，但尚不具备直接立项上生产或采购落地的条件。建议**有条件推荐——先完成真实垂直场景 POC（含权限与人工金标验收）**，再决定规模化投入。  

**总分：66.5 / 100**

## 三、项目优势分析

### 1. 问题定义精准，直击向量 RAG 短板
- **材料依据**：`lixiang.md` / `PRD.md` 明确“答案不在单一片段而在多跳路径”；典型问题如“A 公司母公司 CEO 曾任职何处”“供应商交集”。
- **价值**：避免把项目做成通用聊天机器人，评审目标可对齐可验收的多跳与可解释性。

### 2. 图谱 + 向量 + Agent 分工清晰，非概念堆砌
- **材料依据**：PRD 四层架构；原则写明“图谱是结构化记忆、Agent 是决策大脑、跳数由 Critic 动态决定”；代码中 LangGraph 状态机已接线。
- **价值**：可解释“为何必须上图”，降低“为了 GraphRAG 而上 GraphRAG”的立项风险。

### 3. Agent 闭环具备可执行机制而非口号
- **材料依据**：`agent/loop.py` Planner→Executor→Critic→Answer；`critic.py` 动作枚举 `sufficient|next_hop|rewrite|give_up`；护栏默认 max_hops=5、token 预算、recursion_limit。
- **价值**：多轮检索与止损有工程把手，利于成本与幻觉控制。

### 4. 可审计推理链契约完整
- **材料依据**：`reasoning_chain_v1.json` / `generation/trace.py`：子问题、工具调用、证据 ID、claim 引用、`answered|partial|no_answer`、诚实兜底文案。
- **价值**：直接服务合规审计与 badcase 归因，是企业场景的差异化能力点。

### 5. 评测体系设计专业，具备止损门禁
- **材料依据**：≥200 条分层（2hop/3hop/open/no_answer）、dev/heldout/guardrail；AC-1~7；G1 Conditional-Go、G2 趋势门禁；badcase 归因分类。
- **价值**：可把项目从“演示正确”推进到“可证伪”，降低沉没成本。

### 6. 工程化骨架扎实
- **材料依据**：Repository 抽象（Neo4j/内存图）、统一 Candidate 契约、CI 覆盖率 ≥80%、阶段计划与 `docs/IMPORTANT.md` 延期总账。
- **价值**：后续垂直场景替换时，可在既有骨架上迭代，而非推倒重来。

### 7. 对自身局限有书面诚实性
- **材料依据**：G1/G2 明确标注 offline 启发式 ≠ 生产 LLM；合成语料 ≠ 正式试点；`IMPORTANT.md` 列出权限、消歧、增量、分诊等延期项。
- **价值**：评审可信任项目方治理意识，便于有条件放行而非盲目否决。

### 8. 混合检索与路径爆炸防护有具体参数
- **材料依据**：`default.yaml` 中 max_path_hops=4、每层邻居 50、Top-K 路径 20、高度数阈值等。
- **价值**：说明团队理解图检索工程风险，不仅写“支持多跳”。

## 四、关键短板与不足

### 1. 场景名义与实现域严重偏离
- **问题**：评估对象要求“企业内部知识（制度/合同/工单/流程等）”，实现为合成公司关系宇宙。
- **依据**：`domain_v0.yaml` 仅 Company/Person/Product/Event/Location；`data/pilot` 为脚本生成；`domain_locked: false`、产品签字未完成。
- **影响**：当前效果不能外推到制度条款、审批链、组织职责等企业问题。
- **建议**：立即锁定 1 个真实业务域（建议制度合规或合同审批关系），重做 Schema、语料授权与 50–100 题人工金标。

### 2. 权限与合规能力基本空白（企业一票否决风险）
- **问题**：无文档级/实体级/关系级 ACL，无越权检索防护，无脱敏。
- **依据**：`api/auth.py` 等在延期清单；检索权限过滤「信息不足」；仅有密钥环境变量、租户隔离设想。
- **影响**：一旦接入真实内网文档，存在越权问答与合规事故风险。
- **建议**：POC 起实现“检索前权限过滤 + 答案后二次过滤 + 审计日志”，至少支持文档 ACL 与角色。

### 3. 图谱质量保障仍停留在计划层
- **问题**：实体消歧、跨文档对齐、冲突处理、版本/废止、人工审核 UX 均未落地。
- **依据**：README 明确 POC 不做消歧/增量；质量治理 P3；抽检口径曾为 schema 合法即 correct。
- **影响**：真实文档噪声下，多跳会把错误关系放大为“有路径的错误答案”。
- **建议**：POC 强制：别名词典 + 人工审核队列 + 关系置信度阈值 + 冲突“双版本保留待审”。

### 4. 高准确率证据可信度不足
- **问题**：G1 20/20、G2 +49pp 主要来自离线规则答案器 + 合成金标/确定性路径模板。
- **依据**：`G1_review.md`、`G2_review.md`；badcase 几乎全归 generation；heldout/live 未关。
- **影响**：立项若引用这些数字会造成决策偏差。
- **建议**：门禁只认：真实语料 + live LLM + heldout 人工抽检签字后的数字。

### 5. 复杂度分诊与检索融合未实现
- **问题**：所有请求倾向走 Agentic；RRF 融合未建。
- **依据**：`triage.py`、`fusion.py` 延期；IMPORTANT 写明非融合拼接。
- **影响**：成本/延迟上升；简单事实问也会浪费多轮；向量与图结果冲突无裁决层。
- **建议**：阶段三前做最小规则分诊 + RRF；并定义冲突策略（见下文质询）。

### 6. 企业复杂知识形态覆盖不足
- **问题**：V1 明确不做多模态表格图像；制度条款、流程图、审批记录无专门解析。
- **依据**：PRD Out of Scope；接入偏 MD/TXT。
- **影响**：大量企业知识落在表格/条款结构中，图谱抽取会失真。
- **建议**：POC 场景若含制度/合同，优先做结构化段落 + 条款编号锚点，表格先规则抽取。

### 7. 推理链对业务人员可读性与落库不足
- **问题**：链在响应/JSONL 中，缺审计 API 与可视化；开放路径答案偏路径拼接。
- **依据**：P3-AN-01、P4 UI 延期；G1 备注 open 题可读性弱。
- **影响**：审计“能机器读、难业务读”。
- **建议**：输出“业务摘要路径 + 原文条款链接 + 技术路径折叠”双视图。

### 8. 可量化业务收益缺失
- **问题**：有技术指标（+25pp Accuracy），无工时节省、合规差错率、检索失败率等业务 KPI。
- **依据**：立项书/PRD 均未给出业务 ROI 模型——**信息不足**。
- **影响**：难以与投资评审对齐。
- **建议**：POC 绑定 1 个部门基线：人工查资料平均时长、一次答对率、审计抽检通过率。

### 9. 组织与数据关键路径未关闭
- **问题**：试点领域、标注人力、LLM 预算、延迟阈值仍开放。
- **依据**：PRD §9；风险 R5/R6。
- **影响**：工程再完备也会空转。
- **建议**：评审会一次性锁定：领域、数据 owner、标注人天、月 LLM 预算上限。

## 六、技术路线评估

### 1. 知识图谱方案
- **判断**：作为多跳导航的结构化记忆，**方向合理**；当前本体与治理深度**不足以支撑企业内知识全景**。
- **依据**：Schema V0 关系型清晰；抽取带 confidence/source/quarantine；消歧/增量/版本在 P3。
- **风险**：错误入图会被路径检索“正当化”。
- **建议**：按业务对象重做本体（制度条款、流程节点、合同主体、组织岗位）；POC 阶段接受“小而准”图，拒绝“大而脏”图。

### 2. 向量检索方案
- **判断**：**合理且必要**，作为语义召回与图缺失时的兜底。
- **依据**：chunk 策略、Qdrant 抽象、BM25 互补；Baseline 评测已建。
- **风险**：embedding 与 live 向量库在部分环境未跑通，效果证据偏弱。
- **建议**：保持向量为默认 Fast Path；复杂关系题再升 Agentic。

### 3. Agent 方案
- **判断**：**必要且实现度高于同类概念项目**，不是空壳。
- **依据**：LangGraph 循环、工具选择、Memory 去重、护栏、推理链落点。
- **风险**：live Prompt 质量与成本未充分压测；分诊缺失。
- **建议**：Planner/Critic 用强模型，工具选择用轻模型；补可观测回放（checkpointer 已有基础）。

### 4. 图谱 + 向量 + Agent 协同
- **判断**：**架构协同合理**，**运行时裁决规则不完整**。
- **依据**：Executor 可多工具；RRF 延期；冲突裁决「信息不足」。
- **风险**：多源证据打架时行为不确定。
- **建议**：写死策略：图路径 + 原文引用一致才标“高置信”；冲突则 `partial` 并并列展示。

### 5. 可审计推理链
- **判断**：**设计优秀，工程闭环未完成**。
- **依据**：Schema 完整；落库/可视化/按 query_id 审计 API 在 P3/P4。
- **风险**：合规场景只能“演示审计”不能“生产审计”。
- **建议**：POC 验收强制：任意答案可点到原文 span；无引用不得出最终结论。

## 七、企业场景适配度评估

| 维度 | 适配判断 | 说明 |
|---|---|---|

## 八、是否真正优于传统向量 RAG

### 1. 是否具备明显优势？
**在材料已验证的问题类型上：是（有条件）**。  
在合成公司关系多跳集上，Agentic 路径相对纯向量 Baseline 有显著 accuracy/recall 优势（interim 20 题 +15pp；G2 dev offline +49pp）。  
**在真实企业制度/合同/工单全集上：信息不足，不能下最终结论。**

### 2. 优势体现在哪些具体问题？
- 跨实体归属链（子公司→母公司→CEO→历史任职）
- 需要图结构交集/路径的问题（供应商重叠、产品关联）
- 需要展示推理路径与引用的审计型问答
- 无答案场景的诚实拒答（设计目标明确，guardrail 集有规划）

### 3. 是否存在“为了图谱而图谱”？
**部分风险存在，但未到空壳营销级别。**  
必要性论证充分（静态检索无法适配动态问题结构），且代码与评测对齐。  
风险在于：用 GraphRAG 品牌覆盖尚未实现的企业治理能力，可能造成**营销叙事超前于工程成熟度**。

### 4. 成本过高但收益有限的风险？
**存在。** 多轮 LLM + 图扩展成本高于 Naive RAG；若 80% 问题本可用向量/关键词解决，而系统一律 Agentic，则 ROI 差。分诊未落地放大该风险。

### 5. 场景分流建议
| 更适合图谱增强 + Agent | 仍适合向量/关键词 RAG |
|---|---|
| 跨组织/跨合同/跨系统的关系与责任链 | 单文档事实查询、定义解释 |
| 需路径解释的合规/风控追溯 | 语义相似的经验检索、FAQ |
| 供应商/股权/任职/审批链等多跳 | 关键词精确匹配的制度条目号查询（可先 BM25） |
| 证据冲突需并列展示 | 开放生成型 brainstorm（不应接企业事实库） |

## 九、落地建议

### 第一阶段：最小可验证场景（4–6 周 POC）

**推荐场景：合同/股权/组织职责类“关系链问答”**（或等价的“制度适用主体 + 审批责任人”）  
**不推荐**一上来做全量企业知识中台。

**推荐理由**：
1. 与现有 Schema 与多跳能力迁移成本最低；
2. 业务价值清晰（查责任链、关联主体）；
3. 易构造 2/3 跳金标与审计路径；
4. 可同时验证权限过滤（合同密级）。

若业务强需求合规条款，则改为**“制度合规问答（条款适用对象 + 责任部门 + 废止版本）”**，但需先扩展本体，周期更长。

**POC 验收指标（建议写进合同/立项决议）**：

| 指标 | 门槛 |
|---|---|
| 真实授权文档 | ≥100 篇（或等价条款单元） |
| 人工金标 | ≥80 题（2hop:3hop:no_answer ≈ 4:3:1） |
| live LLM Accuracy vs 向量 Baseline | ≥ +15pp（heldout） |
| 证据 Recall（≥2 跳） | ≥ 75% |
| 无引用编造率 | 0（抽样） |
| 越权测试 | 无权限文档不得出现在证据与答案 |
| 延迟 | 约定 P95（可先放宽到 15s，再收敛 8s） |
| 单题成本上限 | 可配置并熔断 |

### 第二阶段：能力建设（MVP，6–10 周）
- 业务域本体 V1（含别名、生效/废止时间）
- 文档解析（条款编号、表格关键字段规则抽取）
- 实体对齐 + 审核队列
- 文档 ACL 权限过滤贯穿检索与生成
- 推理链业务视图 + 按 query_id 落库
- 200+ 金标（dev/heldout）与一键回归
- 最小分诊 + RRF

### 第三阶段：规模化推广
- 多业务域扩展与跨域实体对齐
- 自动化增量更新与图谱健康度周报
- 运维监控、预算三级硬上限、SSE
- 用户反馈→badcase→策略变更治理
- 模型与图谱联合优化（学习型 rerank 可选）

## 十二、最终结论

### **有条件推荐，需先完成 POC 或补充材料**

**理由（事实层）**：  
- 技术路线自洽，Agent 闭环、推理链、评测门禁、工程骨架均有可核验实现；  
- 已证明在**受控合成多跳任务**上可显著优于纯向量 Baseline；  
- 企业刚需能力（真实域、权限合规、消歧增量、live 效果、业务可读审计）大量仍为计划或延期。

**理由（判定层）**：  
- **值得继续研发投入**，不宜因“GraphRAG 概念泡沫”一刀切否定；  
- **不值得**以当前成熟度直接按生产平台立项、全公司推广或等价采购交割；  
- 正确路径是：**收窄范围 → 真实数据 POC → 权限与审计硬门槛 → heldout/live 验收 → 再扩大投入**。

**建议决议表述（可供会议纪要）**：  
> 原则同意沿 Agentic GraphRAG 路线推进；批准为期 4–6 周的垂直场景 POC（含权限过滤与人工金标）；POC 未达约定指标前，不启动规模化资源与采购；对外效果宣传不得引用 offline 合成集数字作为生产能力证明。
